# 性能优化总结

## 总览

本文档总结了本轮在 `exprion-core`、`exprion-engine` 和 `perf-harness` 上完成的性能优化工作。

这轮工作的目标不只是做零散的小优化，而是系统性地降低前端链路中的明显开销、改善运行时执行路径，并补齐较可靠的性能测试基础设施。

## 已完成的优化

### `exprion-core`

#### 编译上下文与变量栈

- 将较重的同步结构替换为更适合单线程解析场景的实现：
  - `RwLock` -> `RefCell`
  - `Arc` -> `Rc`
- 将原本的符号表整理为更直接的变量栈模型。
- 新增更直观的变量栈接口：
  - `push_scope`
  - `pop_scope`
  - `define`
  - `resolve`
  - `resolve_mut`

#### 变量身份与名字处理

- 为 `Variable` 引入了 `name_id`。
- 在编译上下文 / 变量栈中加入了变量名 intern。
- 作用域查找从字符串键逐步切换为 intern 后的 `NameId`。
- `Variable` 的相等与哈希优先使用 `name_id`。
- 增加了 `same_identity()`，显式表示“按 intern 后身份比较变量”。

这意味着变量密集路径，例如：

- 变量替换
- 方程求解
- 变量收集

已经不再完全依赖字符串比较。

#### 词法与语法分析

- `Lexer` 改为使用“原始字符串 + 偏移量”的推进方式，不再每取一个 token 都重建剩余字符串。
- 新增 lexer lookahead 缓存，使 `peek_token()` 后接 `next_token()` 时不重复解析同一 token。
- 新增 `peek_symbol()`，减少 Pratt parser 对 token 的重复拆解。
- 降低了 parser 侧的 token / symbol 重复匹配开销。
- AST 中变量名改为 `Box<str>`，同时保留 token 层的兼容性。

#### 语义分析与 IR 构造

- 收敛了 `semantic_with_ctx()` 中大量重复的样板分支。
- 增加了 `resolve_variable()` / `resolve_variable_with_value()`，统一变量查找与注册流程。
- 降低了以下模块中的重复变量构造和语义包装开销：
  - `semantic/mod.rs`
  - `semantic/variable.rs`
  - `semantic/semantic_ir/mod.rs`
  - `semantic/semantic_ir/numeric.rs`
  - `semantic/semantic_ir/logic.rs`
- 为 bucket 增加了组合 helper，减少反复的 `clone + push` 模式。
- 在 piecewise 路径中对部分 `Vec` 做了容量预分配。

#### 方程模块

- `contains_target()`、线性提取、多项式提取等路径开始优先按变量身份（`name_id`）判断。
- 变量收集去重逻辑拆成两条：
  - intern 变量按 `name_id`
  - 手写/旧变量按名字

### `exprion-engine`

#### JIT 调用接口

- 新增 `calculate_unchecked()`，将安全接口与纯热路径执行分开。

#### Lowering / 后端耦合

- 引入了面向后端的 lowered IR：
  - `LoweredNumericExpr`
  - `LoweredLogicalExpr`
- 后端不再直接消费前端的语义数值树。
- 变量会在 lowering 早期直接转换为参数槽位索引。

这使后端逻辑更简单，也为后续继续做 JIT 优化打下了更清晰的边界。

#### 幂运算 lowering

- 为常见幂形式做了专门化：
  - `^0`
  - `^1`
  - `^-1`
  - `^2`
  - `^3`
  - `^4`
  - 一般整数幂通过指数平方法展开
  - `^0.5` -> `sqrt`
  - `^-0.5` -> `1/sqrt`

这些场景不再一律走通用 `powf`。

#### Piecewise lowering

- 在 piecewise lowering 前先做常量条件剪枝。
- 从纯 `select` 链切换为基于 basic block + phi 的 piecewise lowering。

#### 逻辑与关系 lowering

- 为以下情况增加了常量短路：
  - 逻辑 `and`
  - 逻辑 `or`
  - 常量对常量的数值关系表达式

#### 编译缓存

- 增加了进程内 JIT compile cache。
- 增加了 cache hit / miss 统计接口。
- harness 输出中已能显示 cache 命中情况。

### `perf-harness`

#### 文件结构

- 输入用例位于 `input/`
- 输出轮次结果位于 `output/`
- 独立维护一个 `best` 文件

#### harness 结构

- 将原本过长的单文件拆分为：
  - `main.rs`
  - `model.rs`
  - `bench.rs`
  - `report.rs`
  - `storage.rs`

#### benchmark 模式

- 增加了多种执行模式：
  - `all`
  - `build_only`
  - `compile_only`
  - `execute_only`

这样可以更干净地隔离不同阶段成本。

#### 结果管理

- 输出文件使用时间戳命名。
- 单独维护 `*_best.toml`。
- 当当前结果优于 best 时，会自动更新 best 文件。

#### 统计

- 增加了：
  - `mean`
  - `median`
  - `min`
  - `max`
  - `jitter_pct`
- 增加了 cache hit / miss 输出。
- 保持了对旧输出格式的一定兼容。

## 当前状态

到目前为止：

- `core` 中明显的同步开销已经被移除
- 变量身份已经 intern 化并进入热路径
- lexer / parser 前端开销已经明显下降
- semantic 到 backend 的耦合已经通过 lowered IR 变得更清晰
- JIT 的幂运算处理已经明显特化
- piecewise lowering 已经更接近正常控制流
- harness 已经比初始版本实用得多

## 为什么这是一个合适的阶段性停点

这一轮优化已经从“低风险小修补”进入了“真实结构优化”阶段。

继续立即往下改当然可以，但边际收益已经明显低于最开始的那批优化。

接下来更重要的是先确认：

- 你最在意的是前端构建延迟
- 还是 JIT 编译延迟
- 还是稳态执行延迟

如果不先明确这一点，后续优化会越来越难评估是否真的值得。

## 建议的下一步

### 如果更重视执行性能

继续在 `exprion-engine` 上推进，重点方向包括：

- lowered IR 的进一步形状优化
- relation / logical lowering
- piecewise 的 heuristic
- 常见数学模式的专门 lowering

### 如果更重视编译延迟

继续扩展 compile cache：

- 更稳定的 cache key
- 更清晰的缓存生命周期
- 更强的可观测性

### 如果更重视长期架构

可以考虑：

- 更早在前端链路中使用 intern/id
- 更激进的 IR normalization
- 如果 profiling 足够明确，再评估 arena / 节点共享结构

## 建议的基线维护方式

在后续实际 benchmark 中，建议：

1. 保留少量具有代表性的 benchmark 表达式。
2. 评估 JIT 执行变化时优先使用 `execute_only`。
3. 把 `best` 文件当作长期参考，但不要只看单次跑分。
4. 如果 `jitter_pct` 很高，就把这次结果视为“不足以支持强结论”的样本。
